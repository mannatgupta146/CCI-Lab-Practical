Digital Forensics Lab Guide

Practical 1: Network Traffic Analysis (Wireshark)

Objective

To capture, analyze, and interpret network traffic using Wireshark, enhancing understanding of network protocols and identifying potential anomalies.

Tools Explained

Wireshark

This is the main tool. It's a "network protocol analyzer" (or "packet sniffer"). It captures all network data passing through your computer's network card (Wi-Fi or Ethernet) and displays it in a human-readable format. You use it to see exactly what your computer is saying to the internet.

Npcap / WinPcap

This is the driver that Wireshark uses to actually *capture* the packets. You usually install it along with Wireshark. You don't interact with it directly.

Detailed Procedure

  1. Install Wireshark: Download and install Wireshark from its official website (wireshark.org). Follow the installation instructions, ensuring you also install Npcap (or WinPcap).
  2. Open Wireshark: Launch Wireshark. You will see a list of available network interfaces.
  3. Select Network Interface: Identify your active network interface (e.g., "Wi-Fi" or "Ethernet"). Double-click on it to start capturing.
  4. Start Capture: Wireshark will immediately begin capturing all network traffic.
  5. Perform Network Activity: While Wireshark is capturing, open your web browser and:
    • Visit a website (e.g., google.com, example.com).
    • Perform a search.
  6. Stop Capture: Go back to Wireshark and click the red square "Stop" button.
  7. Analyze DNS: In the "Filter" bar, type dns and press Enter. Find a DNS query from your computer. Select it and look in the "Packet Details" pane to see what domain was queried.
  8. Analyze TCP Handshake: Clear the filter. Type tcp. Look for the three-packet sequence: SYN, SYN-ACK, ACK. This is the 3-way handshake.
  9. Analyze HTTP Traffic: Clear the filter. Type http. Find an HTTP GET request. Right-click it and select "Follow" > "TCP Stream". This will show you the unencrypted web page data.

Key Concepts & Terms

  • Packet: The smallest unit of data on a network.
  • Protocol: Rules for communication (e.g., TCP, UDP, HTTP, DNS).
  • TCP 3-Way Handshake: The SYN, SYN-ACK, ACK process used to establish a reliable connection.
  • Display Filter: The filter bar in Wireshark used to *find* packets *after* they are captured (e.g., http).
  • IP Address: The "street address" for a device on the internet.

Expected Output

  • Captured network packets in Wireshark.
  • Identification of Source/Destination IP addresses and Port numbers.
  • Analysis of protocols like HTTP (GET requests) and DNS (queries).
  • A clear view of the TCP three-way handshake.
Practical 2: Virtualized Environment Setup

Objective

To implement and understand a virtualized environment, enabling VM server consolidation with scalable infrastructure.

Tools Explained

Oracle VirtualBox / VMware Workstation

These are "Hypervisors" (Type 2). A hypervisor is software that lets you create and run virtual machines (VMs). A VM is a complete computer (CPU, RAM, disk) simulated in software, allowing you to run multiple operating systems (like Linux and Windows) on a single physical laptop.

OVA (Open Virtualization Appliance)

This is a single file (.ova) that contains a pre-configured, pre-installed virtual machine. You "import" it instead of having to install an OS from scratch.

Detailed Procedure

  1. Enable Virtualization: Restart your computer and enter the BIOS/UEFI. Find and enable "Intel VT-x" or "AMD-V". This is required for the hypervisor to work.
  2. Install Hypervisor: Install Oracle VirtualBox or VMware Workstation.
  3. Import Appliance:
    • In VirtualBox: Go to File > Import Appliance...
    • In VMware: Go to File > Open...
    • Select the .ova file provided for your lab.
  4. Configure Resources: Before starting the VM, go to its "Settings". You can adjust the allocated RAM, number of CPU cores, and network settings.
  5. Create VM Manually (Alternative): Click "New" and follow the wizard to create a new VM. You will need an .iso file for an operating system (like Ubuntu Server) to install it.
  6. Consolidation: Start multiple VMs at the same time to simulate server consolidation.
  7. Monitor: Use your host's Task Manager (or htop in Linux) to see how the physical CPU and RAM are being shared by the running VMs.

Key Concepts & Terms

  • Hypervisor: The software that manages VMs.
  • Virtual Machine (VM): A software-based computer.
  • Snapshot: A saved state of a VM. You can "roll back" to this state, which is great for safe experimentation.
  • Server Consolidation: The practice of running multiple servers as VMs on a single physical machine to save power and hardware costs.

Expected Output

  • One or more virtual machines running inside VirtualBox or VMware.
  • Demonstration that the VMs can communicate (e.S. ping each other) or are isolated, based on network settings.
  • Monitoring of the host system's resources (CPU/RAM) being used by the VMs.
Practical 3: Deleted File Recovery (Autopsy)

Objective

To extract and analyze deleted files from a disk image using FTK Imager for imaging and Autopsy for analysis.

Tools Explained

FTK Imager

This is an **acquisition** tool. Its *only* job is to create a perfect, bit-for-bit copy of a storage device (like a USB drive or HDD). This copy is called a "forensic image" (e.g., an .E01 file). You *never* work on the original evidence, so you always make an image first.

Autopsy

This is an **analysis** tool. It's a digital forensics platform. You load the forensic image (created by FTK Imager) into Autopsy. Autopsy then scans the image to find deleted files, browser history, emails, images, and other "artifacts."

Detailed Procedure

Part 1: Create Disk Image (with FTK Imager)

  1. Prepare Target: Plug in a USB drive. Add some files (images, text docs). "Permanently" delete one or two (Shift + Delete).
  2. Launch FTK Imager.
  3. Create Image: Go to File > Create Disk Image...
  4. Select Source: Choose "Physical Drive" and select your USB drive.
  5. Add Destination: Click "Add...". Select the image type "E01". Fill in case details.
  6. Save Image: Choose a location on your *computer's* hard drive and give the image a name (e.g., usb_image).
  7. Start: Click "Start". FTK Imager will create the .E01 file and verify its hash.

Part 2: Analyze Image (with Autopsy)

  1. Launch Autopsy.
  2. Create New Case: Click "New Case". Fill in the case name.
  3. Add Data Source: In the wizard, select "Disk Image" and browse to the .E01 file you just created with FTK Imager.
  4. Configure Ingest Modules: Leave the default modules checked (especially "File Type Identification" and "Deleted File Finder"). Click "Next" to start the analysis.
  5. Find Deleted Files: In the "Tree Viewer" on the left, expand "File Types" and click on "Deleted Files". You should see the files you deleted, often marked with a red "X".
  6. File Carving: Autopsy also "carves" for files, meaning it finds files based on their headers (e.g., JPEG) even if the file system record is gone. You can find these in the "File Types" section by extension.
  7. Extract: Right-click any recovered file and select "Extract File(s)" to save it.

Key Concepts & Terms

  • Forensic Image (E01): A read-only, bit-for-bit copy of a drive that includes all data, even deleted files and unallocated space.
  • Hashing (MD5/SHA1): A "digital fingerprint". FTK Imager hashes the original drive and the image. If the hashes match, it proves the copy is perfect.
  • Data Remanence: The concept that data remains on a disk even after being "deleted".
  • File Carving: Recovering files based on their header and footer signatures, not file system metadata.
  • Unallocated Space: Sections of the disk that are not currently holding an active file. This is where deleted files live.

Expected Output

  • An .E01 forensic image file created by FTK Imager.
  • An Autopsy case containing the loaded image.
  • A list of deleted files (with red 'X' icons) found in the "Deleted Files" section of Autopsy.
  • The ability to extract and view the content of a successfully recovered file.
Practical 4: System Log Analysis (Windows & Linux)

Objective

To understand, extract, and analyze system logs from Windows and Linux to identify indicators of unauthorized access.

Tools Explained

Windows Event Viewer (eventvwr.msc)

The built-in Windows GUI tool for viewing log files. The most important log is the **Security** log, which tracks login attempts, account lockouts, and other security-related events.

PowerShell (Get-WinEvent)

A command-line tool for advanced log querying. It's much faster than the GUI for filtering thousands of events.

Linux Log Files (e.g., /var/log/auth.log)

Linux stores logs as plain text files. /var/log/auth.log (on Debian/Ubuntu) or /var/log/secure (on RHEL/CentOS) is the most critical for forensics as it records *all* authentication attempts (logins, sudo commands, SSH failures).

journalctl

The modern Linux command for viewing logs from the systemd journal, which is replacing many of the old text files.

Detailed Procedure

Part 1: Windows Log Analysis

  1. Simulate Failed Login: Go to the Windows login screen and try to log in with a wrong password several times.
  2. Open Event Viewer: Press Win+R, type eventvwr.msc, and press Enter.
  3. Navigate to Security Log: On the left, expand "Windows Logs" and click on "Security".
  4. Find Failed Logon: Look for an event with **Event ID 4625**. This is a "Logon Failure".
  5. Analyze Event: Click on the Event ID 4625. In the "Details" pane, you can see the "Account Name" that failed to log in and the "Logon Type" (e.g., Type 2 = Interactive, Type 3 = Network).
  6. Find Successful Logon: Look for **Event ID 4624** (Successful Logon).

Part 2: Linux Log Analysis

  1. Simulate Failed SSH Login: From another machine (or Putty), try to SSH into your Linux VM with a wrong password.
  2. Open Terminal.
  3. Check auth.log: Use the grep command to search the authentication log. Type:
    sudo grep "Failed password" /var/log/auth.log
  4. Analyze Output: You will see a list of all failed login attempts, including the time, the invalid username, and the source IP address.
  5. Check Recent Logins: Type lastb to see a list of bad login attempts. Type last to see a list of successful logins.
  6. Use journalctl (Modern way): Type journalctl -u sshd to see all logs related to the SSH service.

Key Concepts & Terms

  • Event ID: A number that Windows uses to categorize events. 4625 (Failure) and 4624 (Success) are the most important to remember.
  • Audit Policy: A setting in Windows that determines *what* gets logged. For forensics, you must enable "Audit logon events" (Success and Failure).
  • grep: A powerful Linux command for searching text. grep "text" file.log.
  • Log Correlation: The process of combining logs from multiple sources (like a firewall *and* a Windows server) to get a full picture of an attack.

Expected Output

  • Identification of Event ID 4625 (Logon Failure) in the Windows Event Viewer.
  • Using grep on /var/log/auth.log in Linux to find "Failed password" entries.
  • Ability to state the difference in log locations for Windows (Event Viewer) and Linux (/var/log/).
Practical 5: Mobile Forensics (MOBILedit)

Objective

To demonstrate acquiring data from a mobile device (using MOBILedit) and analyzing the extracted data (using Autopsy), focusing on artifacts like contacts, call logs, and SMS.

Tools Explained

MOBILedit Forensic Express

This is a commercial **mobile data extraction** tool. Its job is to connect to a mobile phone (Android or iOS) and pull off data like contacts, SMS messages, call logs, photos, and application data. It can perform different types of extractions.

Autopsy

In this lab, you are using Autopsy as the **analysis** tool. After MOBILedit extracts the data, it saves it in a folder. You load this folder into Autopsy (as a "Logical File Set") to analyze the data, just like you did with the disk image in P3.

Android USB Debugging

This is a setting in Android's "Developer Options" that must be *enabled* to allow tools like MOBILedit to access the phone's data.

Detailed Procedure

Part 1: Prepare Phone

  1. Enable Developer Options: On the test Android phone, go to Settings > About Phone and tap on "Build number" 7 times.
  2. Enable USB Debugging: Go to the new "Developer Options" menu and toggle "USB debugging" ON.
  3. Populate Data: Add some dummy contacts, send a few test SMS messages, and make a test call.

Part 2: Extract with MOBILedit

  1. Launch MOBILedit Forensic Express.
  2. Connect Phone: Connect the phone to the PC with a USB cable. On the phone, a pop-up will ask "Allow USB debugging?". Check "Always allow" and tap "OK".
  3. Start Extraction: In MOBILedit, select the device and choose the type of extraction (e.g., "Logical Acquisition").
  4. Select Data: Check the boxes for the data you want to extract (Contacts, SMS, Call Logs, Photos).
  5. Export: After extraction, export the findings. Choose an "Export" or "Report" option that saves the data to a folder (e.g., as CSV files, XML files, or an HTML report).

Part 3: Analyze with Autopsy

  1. Launch Autopsy and create a new case.
  2. Add Data Source: This time, select "Logical Files" (not "Disk Image").
  3. Select Folder: Browse and select the folder that MOBILedit exported its data to.
  4. Start Ingest: Autopsy will scan the folder.
  5. Analyze Artifacts: In the Autopsy tree viewer, look under "Data Artifacts". You should see parsed sections for "Call Logs", "Contacts", and "SMS/MMS Messages" that Autopsy found and organized for you.

Key Concepts & Terms

  • Logical Extraction: Pulls data from the *file system*. It's like asking the phone's OS, "Please give me all contacts, SMS, etc." It's easy but doesn't get deleted data.
  • Physical Extraction: A bit-for-bit copy of the phone's *entire* memory chip. It's much harder to do but gets everything, including deleted data.
  • Artifacts: The pieces of user-generated evidence, like a contact, an SMS message, a call log entry, or a GPS location.
  • EXIF Data: Metadata stored inside a photo file (.jpg) that includes the timestamp, phone model, and sometimes the GPS location of where the photo was taken.

Expected Output

  • A report or folder of extracted data from MOBILedit.
  • An Autopsy case with the extracted data loaded as a "Logical File Source".
  • The ability to view specific artifacts (e.g., SMS messages, Call Logs) in the "Data Artifacts" section of Autopsy.
Practical 6: Cyber Attack Analysis (Wireshark)

Objective

To analyze a pre-captured network traffic (pcap) file representing a cybercrime, identify attack vectors, and reconstruct the events.

Tools Explained

Wireshark

In this lab, you are not capturing live traffic. You are using Wireshark to *open and analyze* a .pcap file. A .pcap (packet capture) file is a recording of network traffic from a past event, in this case, a cyber attack.

Detailed Procedure

  1. Open PCAP: Launch Wireshark. Go to File > Open and select the .pcap file provided for the lab.
  2. Initial Assessment:
    • Go to Statistics > Protocol Hierarchy. This shows you all the protocols in the capture. Look for anything unusual (e.g., lots of FTP, Telnet, or IRC).
    • Go to Statistics > Conversations. This shows who is talking to whom. Sort by "Packets" or "Bytes" to find the "noisiest" (most active) conversations, which are often the attacker and victim.
  3. Hunt for Port Scans:
    • Apply the filter tcp.flags.syn == 1 && tcp.flags.ack == 0.
    • If you see *one* source IP sending packets to *many different ports* on a single destination IP, this is a clear sign of a **SYN Port Scan**.
  4. Hunt for Unencrypted Credentials:
    • Filter for ftp. Look for packets with USER and PASS in the info. Right-click one and "Follow > TCP Stream". You will see the password in plain text.
    • Do the same for telnet.
  5. Hunt for Web Attacks:
    • Filter for http. Look for unusual GET requests (e.g., GET /../../etc/passwd) which could be a Directory Traversal attack.
  6. Reconstruct Timeline: Use the packet timestamps and your findings to build a story (e.g., "1. Attacker (1.2.3.4) performed a port scan. 2. Found FTP open. 3. Brute-forced the password. 4. Logged in and downloaded a file.").

Key Concepts & Terms

  • PCAP (Packet Capture): A file that contains a recording of network packets.
  • Port Scan: An attack used to discover which services (ports) are open on a target machine.
  • Follow TCP Stream: A Wireshark feature that reassembles all the packets from a single TCP conversation and shows you the data in a readable format. This is essential for reading unencrypted protocols like HTTP, FTP, and Telnet.
  • Protocol Hierarchy: A Wireshark statistic that gives you a percentage breakdown of all protocols in the capture.

Expected Output

  • Identification of the Attacker's IP and Victim's IP from the "Conversations" window.
  • Evidence of a specific attack (e.g., a port scan, a plain-text password from an FTP stream).
  • A summary of the attack timeline.
Practical 7: Email Threat Analysis (Phishing)

Objective

To demonstrate and understand how advanced email threat detection techniques identify phishing, malware, and business email compromise (BEC).

Tools Explained

Social-Engineer Toolkit (SET)

A tool (often in Kali Linux) used to simulate and execute social engineering attacks. It can be used to craft a convincing phishing email and clone a website for credential harvesting.

Mail Server / Client (Thunderbird, Outlook)

You need a mail client (like Thunderbird) to *receive* the test phishing email and a mail server to *send* it.

Detailed Procedure

  1. Set Up Environment: You need an "Attacker" VM (like Kali) and a "Victim" VM (like Windows) with a mail client.
  2. Craft Phishing Email (Attacker):
    • Use a tool like SET or a simple Python script to create a fake email.
    • The email should have red flags, such as:
      • A "spoofed" (fake) sender address.
      • A sense of urgency (e.g., "Your account is locked!").
      • A malicious link that *looks* real but points somewhere else.
      • A malicious attachment (e.g., a .zip or .docm file).
  3. Send Email: Send the email from the attacker to the victim.
  4. Analyze Email Header (Victim):
    • In the victim's mail client (Thunderbird/Outlook), open the email and find the option to "View Source" or "View Message Header".
    • Trace the Path: Look at the Received: headers. Read them from **BOTTOM to TOP** to see the true path the email took.
    • Check Authentication: Look for a header called Authentication-Results. Check the status of SPF, DKIM, and DMARC. If they say "fail" or "none", it's a huge red flag that the sender is spoofed.
  5. Analyze Attachments/Links: (CAREFULLY!) Do not click the link. Instead, hover over it to see its true destination. If it's an attachment, you would normally submit its hash to VirusTotal.

Key Concepts & Terms

  • Phishing: An email scam to steal credentials (username/password).
  • BEC (Business Email Compromise): A targeted scam that impersonates a high-level executive (like the CEO) to trick an employee into sending money.
  • Email Header: The metadata of an email that shows the true sender, path, and authentication results.
  • SPF (Sender Policy Framework): An email authentication record that prevents spammers from sending messages on behalf of your domain.
  • DKIM (DomainKeys Identified Mail): Adds a digital signature to emails to verify they are legitimate.
  • DMARC: A policy that tells a receiving server what to do if SPF or DKIM fails (e.g., "reject" the email).

Expected Output

  • A received phishing email in the victim's inbox.
  • A clear analysis of the email's header.
  • The ability to point to the Authentication-Results (SPF/DKIM/DMARC) header and explain what "fail" means.
  • The ability to trace the Received: headers from bottom to top.
Practical 8: Browser Privacy Analysis

Objective

To observe and analyze the types of data collected by websites and how browser privacy tools can limit this collection.

Tools Explained

Browser Developer Tools

Every modern browser (Chrome, Firefox) has built-in "Dev Tools". You access them by pressing **F12**. For this lab, you'll use two tabs:

  • Network Tab: Shows you every single request the website makes (to load images, scripts, ads, trackers).
  • Application Tab (or Storage): Lets you see the website's storage, including Cookies and Local Storage.

Privacy Badger / uBlock Origin

These are browser extensions that identify and block third-party trackers, ads, and cookies.

Detailed Procedure

Part 1: Baseline (No Protection)

  1. Open your browser (e.g., Chrome) in a new Incognito window.
  2. Press F12 to open Developer Tools.
  3. Click the "Network" tab. Check the "Disable cache" box.
  4. Click the "Application" tab (or "Storage" in Firefox).
  5. Visit a Website: Go to a tracking-heavy website (like a major news site).
  6. Observe Network: In the "Network" tab, watch as *hundreds* of requests are made to third-party domains (ad networks, analytics).
  7. Observe Storage: In the "Application" tab, look at "Cookies" and "Local Storage". See all the trackers setting data. Note the total number of cookies.

Part 2: With Protection

  1. Install a privacy extension (like uBlock Origin or Privacy Badger).
  2. Open a new Incognito window (ensure the extension is allowed in Incognito).
  3. Open Dev Tools (F12) again.
  4. Visit the SAME Website.
  5. Observe Network: Look at the "Network" tab. You will see many requests are now blocked or are red. The page will load faster.
  6. Observe Storage: Go to the "Application" tab and check "Cookies". You will see significantly *fewer* cookies have been set.

Key Concepts & Terms

  • Cookie: A small piece of data a website stores in your browser to remember you (e.g., for login sessions or ad tracking).
  • Third-Party Cookie: A cookie set by a domain *other than* the one you are visiting. This is the main tool of ad trackers.
  • Local Storage: A newer, larger storage area in the browser that websites can use.
  • Browser Fingerprinting: A technique used to identify you *even without cookies*, by collecting details about your system (screen resolution, fonts, browser version, etc.).

Expected Output

  • Screenshots of the "Network" tab before and after, showing a reduction in total requests.
  • Screenshots of the "Application" tab before and after, showing a reduction in the number of cookies.
  • A clear comparison of the data collected with and without privacy tools.
Practical 9: Data Remanence (Recuva & TestDisk)

Objective

To empirically analyze data remanence and the effectiveness of data recovery tools, specifically Recuva (file recovery) and TestDisk (partition recovery).

Tools Explained

Recuva

This is a **file recovery** tool. It is used to find and recover *individual files* that have been deleted from a working partition. It's simple, user-friendly, and works by scanning for file system records that have been marked as "deleted".

TestDisk

This is a **partition recovery** tool. It is a powerful, command-line-based tool used to fix *entire partitions* that have been deleted or corrupted. If your USB drive suddenly shows up as "Unallocated" or "RAW", TestDisk is the tool you use to find the lost partition table and re-write it.

Detailed Procedure

Part 1: File Recovery (Recuva)

  1. Prepare Drive: Take a USB drive. Format it. Create a folder and put some files (images, docs) inside.
  2. Delete Files: Select the files and press Shift + Delete to permanently delete them.
  3. Scan with Recuva:
    • Open Recuva.
    • Select the drive (your USB) and the file type you're looking for (or "All Files").
    • Click "Scan". Recuva will find the deleted files.
  4. Recover: Check the box next to the file you want and click "Recover". **IMPORTANT:** Save the recovered file to your *computer's* hard drive, NOT back to the USB drive (this could overwrite other deleted data).
  5. Test "Quick Format": Quick-format the USB drive, then run Recuva again. It will *still* find the files.

Part 2: Partition Recovery (TestDisk)

  1. Delete Partition: Go to Windows "Disk Management". Right-click your USB drive's partition and select "Delete Volume". The drive will now show as "Unallocated Space".
  2. Launch TestDisk: Run testdisk_win.exe (it's a command-line interface).
  3. Create Log: Select "Create" for a new log file.
  4. Select Disk: Use the arrow keys to select your USB drive (NOT your main C: drive!) and press Enter.
  5. Select Partition Type: Choose "[Intel/PC partition]".
  6. Analyze: Select "[Analyse]" and press Enter.
  7. Quick Search: On the next screen, select "[Quick Search]". TestDisk will scan the unallocated space.
  8. Find Partition: TestDisk should find your "lost" partition. Press "P" to list the files and verify it's the right one.
  9. Write Partition: Press Enter, then select "[Write]" to restore the partition table. Confirm with "Y".
  10. Reboot: Safely eject and re-plug the USB, or reboot your PC. The partition will be back.

Key Concepts & Terms

  • Data Remanence: The persistence of data on storage even after deletion.
  • Quick Format: Simply erases the "table of contents" (file system index) of a drive. The data is still there. Recuva can recover from this.
  • Full Format: Erases the index *and* overwrites the entire drive with zeros. Data is not recoverable after this.
  • Partition Table: The map at the very beginning of a drive that tells the OS where each partition (e.S. C:, D:) starts and ends. TestDisk fixes this map.

Expected Output

  • A file (e.g., a .jpg) successfully recovered using Recuva.
  • A USB drive partition, previously "Unallocated", successfully restored and made visible in "This PC" using TestDisk.
Practical 10: Forensic Report Writing

Objective

To develop comprehensive forensic reports for phishing incidents, using forensic tools for analysis and a word processor for documentation.

Tools Explained

EnCase / Autopsy

These are the **analysis** tools. You use them to perform the technical investigation, find the evidence (like the phishing email, the malicious file, the browser history of the victim clicking the link), and take screenshots.

Microsoft Word / Word Processor

This is the **reporting** tool. This is where you *present* your findings. A forensic report is a formal, legal document. It must be clear, concise, and accurate. Your technical skills are useless if you can't *explain* what you found in a report.

Detailed Procedure (Report Structure)

A good forensic report has a standard structure. When writing your report, you must include these sections:

  1. Case Details:
    • Case Number, Date of Report, Investigator Name.
  2. Executive Summary:
    • A *one-paragraph* summary for a non-technical manager.
    • Example: "On Nov 12, 2025, an employee received a phishing email. They clicked the link, and malware was installed. We analyzed the system and found evidence of data exfiltration. The system has been contained."
  3. Methodology:
    • Describe the tools you used (e.g., "FTK Imager v4.2.0, Autopsy v4.19.0").
    • Describe your process (e.g., "A forensic image was acquired, and hashes were verified. The image was analyzed...").
  4. Findings & Analysis (The most important part):
    • Present your evidence logically.
    • Show the phishing email header (and explain the SPF/DKIM failure).
    • Show the browser history of the click.
    • Show the malicious file found on disk (with its hash).
    • Show the network logs (from Wireshark/Autopsy) of the C2 communication.
    • Use screenshots from your analysis tools.
  5. Conclusion:
    • A brief summary of your findings (e.g., "The evidence confirms the system was compromised via phishing...").
  6. Recommendations:
    • Actionable steps to fix this. (e.g., "1. Re-image the workstation. 2. Reset the user's password. 3. Implement employee phishing training.").
  7. Appendices (Chain of Custody):
    • Include your **Chain of Custody** form.

Key Concepts & Terms

  • Chain of Custody: The *most important* concept in all of forensics. It is a log that documents *who* handled the evidence, *when* they handled it, *where* it was stored, and *why* they handled it. A broken CoC makes your evidence inadmissible.
  • Indicators of Compromise (IOCs): The "clues" of an attack. These are things like the attacker's IP, the hash of the malware file, or the phishing email's subject line.
  • Timeline Analysis: The process of putting all the evidence in chronological order (using timestamps) to build a narrative of the attack.

Expected Output

  • A complete, multi-page forensic report in .doc or .pdf format.
  • The report must be structured with the sections listed above (Summary, Methodology, Findings, etc.).
  • The report must contain screenshots from your analysis tools (like Autopsy) as evidence.
